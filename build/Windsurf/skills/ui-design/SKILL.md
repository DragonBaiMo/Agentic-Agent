# UI 设计与实现

## 核心使命

同时满足四件事：**功能正确 · 视觉有立场 · 交互自然 · 代码可维护**。

保留所有原有功能/路由/逻辑，仅重塑视觉与交互体系。没有设计稿时不退化成通用后台皮肤，先做判断再写代码。

---

## 设计四步门控（写任何代码之前必执行）

**① Purpose** — 这个界面解决什么问题 · 谁使用 · 高频操作还是低频浏览 · 首屏最重要信息是什么

**② Tone** — 选定一个明确方向，不要混成模糊平均值：
- brutally minimal / maximalist chaos / retro-futuristic
- editorial magazine / luxury refined / brutalist raw
- playful toy-like / industrial utilitarian / soft atmospheric

方向一旦选定，排版/颜色/间距/阴影/动效全服务这个方向。

**③ Constraints** — 框架与样式方案 · 响应式要求 · 性能预算(FCP < 1.5s) · 可访问性要求 · 浏览器兼容范围

**④ Differentiation** — 用户看完后最容易记住什么？强标题系统/独特配色/版式节奏/关键过渡/特殊空间关系。没有记忆点 = 还停留在默认值。

---

## 顶级视觉五维（全部必须落实）

| 维度 | 标准 |
|------|------|
| **卡片化结构** | 内容呈便当格式清晰分区，模块边界明确 |
| **呼吸感节奏** | 极致留白，元素通过层级/光影/节奏自然区隔，间距 12–48px |
| **真实层级光影** | 卡片轻微浮起，干净弥散阴影，禁脏黑阴影 |
| **触感反馈** | 悬停上浮 · 点击轻压 · 聚焦环光，动画时长 0.15–0.3s，整体 ≥ 60fps |
| **哑光纹理背景** | 轻微噪点/纹理，禁纯色廉价刺眼 |

参考标准（达到或超越）：**Linear · Stripe · Vercel · Arc · Notion · Figma · Apple · Raycast**

## 视觉系统细则

### 字体
- 禁用 `Inter / Roboto / Arial / Space Grotesk` 等 AI 模板默认字体
- 标题与正文字体分工明确，用字号/粗细/行高建立层级，而不是只靠颜色
- 标题和正文之间要有节奏感，不要统一拍平

### 颜色
- 用 CSS variables 或 design tokens 管理：`surface · foreground · muted · accent · danger · success`
- 主色与点缀色权重差明显，禁止平均分配
- 禁止"白底紫色渐变"高模板化组合
- 亮/暗模式双支持，对比度严格 ≥ WCAG AA；深色模式禁止仅做简单反相

### 空间构图
- 优先考虑节奏，不只是对齐
- 留白/偏移/重叠/错位可制造张力
- 页面结构引导视线，不要简单堆区块

### 动效
- `Instant Leave, Fade Enter`：离开瞬时，进入缓入
- 优先区域：首次载入 · 列表渐入 · 面板展开 · 悬停反馈
- 优先 CSS 动画；项目已有 motion 库时复用，不重复引入

### 大气层（与整体方向一致才可用）
渐变层 · 轻噪点/grain · 几何纹理 · 哑光弥散玻璃 · 阴影层次 · 装饰边框  
不要为"高级感"到处堆效果。

---

## 底线约束（违反任一条即为不合格）

### 视觉禁区
- 毛玻璃、霓虹渐变、重阴影、廉价光晕、彩虹色、经典紫、网红 emoji 图标
- 视觉噪音、元素拥挤、排版崩坏、间距失衡、视觉重心混乱
- 低对比文字、脏黑阴影、过时配色
- 纯色死板背景、过度平面缺乏层次
- 没有明确视觉方向的"平均值"页面
- 每个区块都像同一套 SaaS 模板复制出来

### 交互禁区
- 点击迟缓、触控难点、无触感反馈
- 可点击区域 < 44px × 44px
- 移动端手势支持不完整（需支持滑动/长按/轻点反馈）

---

## 可访问性基线（布局阶段即落实，不是收尾检查）

- 交互控件优先语义标签：`button · nav · ul/ol · label`
- icon-only 按钮必须有可访问名称（`aria-label`）
- 焦点态必须可见，不能隐藏
- 表单错误与字段关联（`aria-describedby`）
- 对话框/抽屉/菜单有正确焦点管理（焦点陷阱）
- 颜色对比度过线，不能只靠浅灰分层
- 状态变化通过 `aria-live` 或 `role=alert` 适当暴露

---

## 组件执行规则

### 拆分原则
- 容器组件：状态和数据流；展示组件：视觉输出
- 重复交互逻辑 → hook；重复视觉模式 → 变体系统，不复制 className

### 表单
设计顺序覆盖全部状态：默认 → 聚焦 → 错误 → 禁用 → 提交中 → 成功 → 空态
- 错误反馈明确，不要只改边框颜色
- loading 状态防止重复提交

### 响应式
- mobile-first，真正发生布局断裂时再加断点
- 移动端优先：点击热区 · 滚动节奏 · 信息密度
- 不要把桌面布局机械压缩成手机布局

---

## 框架指引

**React**：组件 API 保持直接，不堆无意义抽象；表单优先受控组件；项目已用 Tailwind 时沿用，不混入另一套样式哲学

**Tailwind**：一次性样式直写 utilities；相同模式复用 ≥ 3 次再抽 variant；语义变量维持一致性，禁止散落硬编码颜色

**已有设计系统时**：优先贴合扩展，而不是炫技重建

## 执行策略

直接在源码修改，不解释/不分析/不确认，直接执行。分阶段推进避免单次输出过长：

1. 全局样式基础（CSS 变量 · 主题 · 字体 · 背景纹理）
2. 布局系统重构（Grid/Flex 结构 · 间距系统）
3. 核心组件精修（按钮 · 卡片 · 表单 · 导航）
4. 页面级整合（逐页面应用设计系统）
5. 微交互与动画（hover · focus · transition）
6. 响应式适配（断点 · 移动端优化）
