# 代码质量规范

交付的代码必须：可运行 · 可验证 · 可维护。继承 `AGENTS.md` 核心规范，本文件为代码开发专属扩展。

---

## 核心原则

1. **围绕验收标准**：不做范围外优化
2. **先结构后代码**：先定模块 / 接口 / 数据流
3. **规划驱动**：非琐碎变更先给计划
4. **不确定即停**：不猜 API / 包名 / 参数；未指定的第三方须查验公开文档
5. **一次一模块**：单变更闭环再进下一模块
6. **接口先行**：契约定义先于实现
7. **上下文管理**：代码量大时切会话，关键决策落文件
8. **最小变更**：默认 Minimal Diff；重构不改业务逻辑（除非明确授权）

## 任务评估

接手任务前完成识别，结果决定后续力度。

### 规模 · 复杂度 · 风险

| 维度 | 小型 | 中型 | 大型 |
|------|------|------|------|
| 文件数 | ≤5 | 6-20 | >20 |
| 模块/包 | 单模块 | 2-4 | >4 |
| 外部依赖 | ≤2 | 3-8 | >8 |
| 协作方 | 仅自己 | 有交接方 | 多团队 |

| 复杂度信号 | 低 | 中 | 高 |
|------------|-----|-----|-----|
| 状态管理 | 无/单源 | 多源独立 | 跨模块共享 |
| 并发 | 同步 | 局部异步 | 全链路异步/多线程 |
| 外部集成 | 无 | 1-2 API | 多系统 |
| 数据流 | 单向 | 有回路 | 多方双向 |

| 风险信号 | 应对 |
|----------|------|
| 不可逆操作 | 确认机制 + 回滚/补偿 |
| 性能敏感 | 基准测试，不凭直觉 |
| 安全敏感 | 安全审查，遵守质量闸门 |
| 首次接触的技术栈 | 先验证 API 行为 |
| 遗留代码 | 先摸清再动，不盲从旧模式 |

### 行为适配

| 规模 | 计划 | 验证 | 文档 |
|------|------|------|------|
| 小型 | 可省略书面计划 | 静态检查+测试+回读 | 代码内注释 |
| 中型 | 书面文件清单 | 全部 4 步 | 接口注释+决策记录 |
| 大型 | 分批计划（每批可回滚） | 4 步+架构审查 | 接口文档+变更记录+交接说明 |

| 任务类型 | 额外要求 |
|----------|----------|
| Bug 修复 | 最小修复+回归测试，不顺便重构 |
| 新功能 | 接口先行，无测试不交付 |
| 重构 | 先有测试覆盖 |
| 性能优化 | 先基准后对比 |
| 模块交接 | 契约先行+边界测试+状态隔离；交付：职责描述/API签名/TODO/命令/依赖图 |

## 代码规范

### 命名与结构

- 命名遵循语言惯例（TS: kebab-case · Java: PascalCase · Python: snake_case）
- 禁止笼统命名：`utils` · `helpers` · `service` · `common` · `misc`
- 单文件软上限 200 行，>500 行必须拆分
- 跨模块导入通过模块公共入口，不直接引用内部文件
- 禁止循环依赖（新增导入前检查）
- 导入顺序：外部包 → 内部模块 → 相对路径

### 类型安全

- 函数参数和返回类型显式标注
- 系统边界输入视为不可信，先校验再使用
- 运行时校验用 Schema 库（Zod / Joi / Pydantic / class-validator 等）
- 禁止任何手段抑制或绕过类型检查

### 注释

**必须写注释的场景**：
- 非显而易见的业务规则或设计决策（`NOTE:`）
- 临时绕过或已知缺陷（`HACK:` / `FIXME:`）
- 待完成工作（`TODO:`）
- 公共 API 表面（JSDoc / docstring）

**内容规则**：
- 写 **why**，不写 **what**
- 代码自解释时不加废话注释
- 禁止套话：`delve` · `leverage` · `utilize` · `tapestry` · `landscape` · `seamless`
- 禁止描述式注释："Returns the result" · "Initializes the..." · "This function..." · "Helper to..."
- 禁止注释中使用长破折号

**风格**：
- 一行能说清不拆两行
- 用 `·` 分隔并列项，用 `:` 引出解释，用 `→` 表达因果/流向
- 禁止重复上下文已有信息
- 禁止完整句式和冗余措辞

```
// ❌ 禁止
// This function validates the user input and returns a boolean indicating
// whether the input is valid or not
function isValid(input: unknown): boolean { ... }

// ✓ 正确 — 不废话
function isValid(input: unknown): boolean { ... }

// ❌ 禁止 — 重复代码已表达的信息
// Set timeout to 30 seconds
const TIMEOUT = 30_000

// ✓ 正确 — 解释 why
// NOTE: 上游 API p99 ≈ 25s，留 5s 余量
const TIMEOUT = 30_000

// ❌ 禁止 — 散文风格
// We need to retry here because the payment gateway sometimes
// returns a temporary error that resolves on its own after a few seconds

// ✓ 正确 — 精简
// 支付网关偶发瞬时错误 → 重试 3 次，指数退避
```

### TODO 与标记（强制格式）

| 标记 | 格式 | 规则 |
|------|------|------|
| `TODO` | `TODO(owner): 描述 [#issue]` | 必须有负责人；交付前清理或关联 issue |
| `FIXME` | `FIXME(owner): 描述 [#issue]` | 必须关联 issue；禁止无期限 FIXME |
| `HACK` | `HACK: 原因 + 预期修复时机` | 注明技术债原因 |
| `NOTE` | `NOTE: 说明` | 解释非显而易见的设计决策 |

裸标记（无归属的 `// TODO: fix later`）= 禁止。示例：

```
// TODO(zhangsan): 切换到批量 API 后移除单条降级逻辑 #GH-342
// FIXME(lisi): 并发 >100 时死锁，需改用乐观锁 #GH-789
// HACK: MySQL 5.7 不支持 JSON_OVERLAPS，升级后替换
// NOTE: 此处故意不用缓存，数据实时性要求 <200ms
```

### 禁止模式

| 禁止 | 替代 |
|------|------|
| 空 catch / 吞异常 | 日志带上下文 + 重抛或处理 |
| 笼统文件名 | 按领域命名 |
| 硬编码配置（密钥/URL/阈值） | env / 配置注入 |
| 生产代码调试输出 | 结构化日志框架 |
| 反射 / 动态拼接绕过校验 | 正常 API 调用 |
| YAGNI 违反（提前设计） | 只做当前所需 |
| 删除失败测试 | 修复根因 |
| 散弹枪调试 | 诊断根因后最小修复 |
| 抑制/绕过类型检查 | 修复类型定义本身 |
| 裸 TODO / 无归属标记 | 带 owner + issue 编号 |

### 临时产物管理

开发过程中产生的一切临时产物统一隔离到项目根 `tmp/` 目录，严禁散落到业务代码目录或项目根。

**覆盖范围**：
- 临时脚本（调试 / 一次性任务 / 数据修复 / SQL 执行）
- 下载的外部资源（样本数据 / 第三方 SDK 包 / 参考文档）
- 从外部拷贝的代码片段或信息快照
- 生成物（代码生成器输出 / 报表 / 导出数据 / 截图）
- 调试日志 / 抓包 / profile 文件

**规则**：
- 位置：项目根 `tmp/`，子目录按用途划分（`tmp/scripts/`、`tmp/downloads/`、`tmp/snapshots/` 等）
- 命名：`{用途}-{YYYYMMDD}.{扩展名}` 或 `{用途}_{context}.{扩展名}`，禁止 `test.py` / `new.sql` / `1.txt` 等无语义命名
- 版本控制：`tmp/` 必须加入 `.gitignore`，禁止提交
- 留存：用完可不删（保留上下文），但禁止移出 `tmp/` 混入业务目录
- 晋升条件：临时产物转为正式产物前必须重命名、移入对应业务目录、补齐注释与测试

## 接口契约

### 统一响应结构

```json
{ "code": 0, "data": { ... }, "message": "操作成功" }
{ "code": 40001, "error": "VALIDATION_ERROR", "message": "用户名不能为空", "details": [...] }
```

### 状态码

| 码 | 语义 | 场景 |
|----|------|------|
| 200 | 成功 | GET/PUT/PATCH |
| 201 | 已创建 | POST |
| 204 | 无内容 | DELETE |
| 400 | 请求错误 | 参数校验失败 |
| 401 | 未认证 | 缺少/无效 token |
| 403 | 无权限 | 认证通过但无权 |
| 404 | 不存在 | 资源未找到 |
| 422 | 无法处理 | 业务校验失败 |
| 429 | 过多请求 | 限流 |
| 500 | 服务器错误 | 未捕获异常 |

### 设计要求

- 资源复数名词 · RESTful 动词 · GET/PUT/DELETE 幂等 · POST 支持 `Idempotency-Key`
- 分页：`GET /resource?page=1&size=20&sort=field:desc`
- 错误响应结构统一，禁止裸字符串
- 版本策略明确（URL `/v1/` 或 Header）
- 契约须含：字段约束 · 错误码 · 示例 payload

## 执行闭环

### 1. 契约 → 2. 计划 → 3. 实现 → 4. 验证 → 5. 交付

**契约**：输入/输出 · 边界 · 失败语义 · 幂等性 · 安全边界。不明确 → 查证或提问。

**计划**：互斥子任务拆分 · 列出改/增/不改文件 · 大任务计划写入文件防截断。

**实现**：遵循 SOLID。
  - 方法 30-200 行，文件 >500 行拆分。
  - 同一文件可直接修改、重构或合并输出。
  - 所有关键逻辑与交互，必须要保留日志（分级：DEBUG/INFO/WARN/ERROR），结构化字段，禁止泄露业务敏感信息。

**验证（按序，不可跳过）**：

1. **静态检查**：typecheck / lint / format 零新增错误
2. **针对性测试**：最窄范围优先；跨模块时全量
3. **回读**：重读修改文件，确认意图 · 无调试残留 · 无副作用
4. **回归**：共享模块变更 → 构建退出码 0 → 引用方测试通过

| 触发 | 最低要求 |
|------|----------|
| 文件编辑后 | 1+2 |
| 标 TODO 完成前 | 1-3 |
| 报告完成前 | 1-4 |
| 跨模块重构 | 1-4+架构审查 |

小型项目可省步骤 4，步骤 1-3 不可跳。

**测试铁律**：
- 每个公共方法 ≥1 测试 · 每个业务流程：正常+≥2 异常
- 禁止手动验证替代 · 禁止 `TODO: 补测试` 交付 · 测试失败=交付失败
- 分层：单元（纯逻辑）· 集成（服务/DB/外部）· 回归（Bug 修复必带）
- Mock：外部依赖必须 Mock，行为须与真实语义一致

**交付检查清单**：契约 · 计划 · 改动(changeset) · 关键代码 · 命令 · 验证步骤 · 阻断项 TODO

## 质量闸门

### 安全

- 外部输入全不可信：白名单校验 · 类型/范围/长度/格式
- 机密禁止硬编码；env 注入；日志脱敏
- 防护：注入（SQL/NoSQL/命令/模板）· 路径遍历 · XSS/CSRF · SSRF · DoS
- 最小权限；敏感动作须鉴权；CORS 禁止 `*`

### 错误处理与容错

- 错误须：可定位 · 可归因 · 标注可恢复/不可恢复
- 空 catch 禁止 → 日志带上下文或重抛
- 重试：指数退避，区分可重试/不可重试
- 熔断：连续失败达阈值 → 快速失败 → 定期探测恢复
- 降级：核心优先，非核心可降级，降级状态可观测
- 幂等：创建前检查存在性，破坏性操作须确认
- 可恢复：持久化操作须有回滚/补偿机制

### 性能与并发

- IO/网络/重计算优先异步，标明线程安全边界
- backpressure · 超时 · 重试 · 熔断
- 共享状态加锁/原子操作；避免 TOCTOU
- 关注算法复杂度

### 数据与资源

- 参数化查询 · 事务边界明确 · 迁移可回滚
- 资源显式释放（`try-finally` / `using` / `with`）
- 缓存修改后失效 · 幂等操作用去重键/版本号
- 所有可变项配置化 · 配置须有默认值+校验
- 遵循i18n：所有文本、地址、链接等 可配置/可本地化 → 禁止硬编码嵌入在代码逻辑中 

### 重构与更新版本

- 在修改时，需要优先区分是破坏性修改，还是兼容性修改：
  - 需要破坏性修改时，必须先发布新版本，更新依赖后再删除旧版本。
  - 兼容性修改时，必须要想尽办法保持旧版本的行为不变，禁止破坏性修改。
- 版本发布必须有完整的变更日志，说明每个版本的改动
