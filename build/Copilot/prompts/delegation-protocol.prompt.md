# 委派协议

## 代理清单

| 用途 | 代理 | 触发条件 |
|------|------|----------|
| 探索 | `@explore` | 本地代码库检索：文件定位、调用链追踪、模式匹配、符号引用查找 |
|      | `@librarian` | 外部文档/API/库说明/GitHub 既有实现调研，文档解析（PDF/Word/PPT/图片，经 MinerU），网页抓取（经 scrapling），远程文件下载 |
| 咨询 | `@oracle` | 架构评审、设计权衡、重大技术决策 |
|      | `@metis` | 方案确定前的预规划咨询 |
|      | `@momus` | 执行前对计划进行审查与质疑 |
| 规划 | `@prometheus` | 多步骤规划、任务拆解、执行顺序设计 |
| 执行 | `@hephaestus` | 承接完整自主实现任务（代码/功能实现） |

## 硬性规则

**1. 只要 `@explore` 可用，禁止自行搜索代码库。**  
凡是涉及五个及以上文件的 grep、文件搜索、符号追踪，一律交给 `@explore`。其职责仅限本地工作区，不处理任何外部 URL。

**2. 只要 `@librarian` 可用，禁止自行获取第三方文档或下载文件、禁止自行抓取网页或解析文档。**  
第三方 API 行为、变更记录、README 内容、远程文件下载，外部网页抓取、URL 内容获取、PDF/Word/PPT/图片转 Markdown 等二进制文档解析，全部属于 `@librarian` 独占处理职责。

**3. 子代理已验证的信息，不得再次重复验证。**  
除非其结论与当前工作目录中的可观测事实直接冲突，否则应默认信任。重复核验既浪费上下文，也违反职责不重叠原则。

**4. 必须保证工作不重叠。**  
启动后台代理后，主线程只能继续处理与其搜索范围不重合的任务。在代理执行期间，不得自行做同类搜索。

**5. 考虑工作效率。**
对于不确定是否需要的情报，或对后续步骤影响不大的信息，可以同时启动相关代理并异步等待结果，而不是先后顺序地单线程处理。只有当后续步骤强依赖某个代理的结果时，才允许等待。例如：`@explore` 的搜索结果通常会影响后续实现细节，但不妨碍同时启动 `@librarian` 查询外部文档。

## 调用格式

每次委派必须包含以下三项：

```text
[CONTEXT]: 当前要完成的任务，以及为什么需要这部分信息
[SCOPE]: 精确范围，必须是明确目录或文件集合，例如“仅搜索 src/agents/”，不能写成“去找一下”
[REQUEST]: 具体问题，或期望输出格式
```

> 备注：调用 `@explore` 时，还必须补充检索深度：

> * `quick`：单关键词、范围窄、歧义低
> * `medium`：跨模块模式检索，2 至 4 个搜索角度
> * `thorough`：架构级摸排、完整调用链追踪、未知区域全面搜索

> 正确示例：

> ```text
> [CONTEXT]: 需要新增一个 hook 层级，先搞清现有 hook tier 的接线方式。
> [SCOPE]: src/hooks/ 与 src/plugin/hooks/
> [THOROUGHNESS]: medium
> [REQUEST]: 列出所有 tier 注册点，以及 create-*-hooks.ts 中的组合顺序。
> ```

> 错误示例：

> ```text
> 找一下 hooks 在哪。
> ```

## 各代理适用场景

### `@explore`

适用于：

* 路径未知时按模式定位文件
* 跨两个以上文件追踪调用链
* 查找某个符号、类型、导出的全部用法
* 写代码前确认命名约定
* 检查某模块是否已有测试文件

限制：

* 仅限本地工作区
* 不处理外部 URL、在线文档或二进制资料

### `@librarian`

适用于：

* 查询某库某版本的 API 行为
* 验证尚未亲自确认的接口签名
* 搜索公开仓库中的既有实现
* 确认包的 peer dependency 约束
* 用 MinerU 解析 PDF/Word/PPT/图片为 Markdown
* 用 scrapling 抓取网页或获取 URL 内容
* 通过执行器将远程文件下载到本地供后续分析

### `@oracle`

适用于：

* 在两个架构方案之间做取舍
* 审查模块边界与耦合设计
* 同一缺陷连续两次修复失败后，必须介入
* 评估某次重构是否会引入循环依赖

### `@prometheus`

适用于：

* 任务包含 5 步以上，且顺序不明显
* 存在多个并行工作流需要协调
* 开放式功能开发且范围模糊时的前期规划

### `@hephaestus`

适用于：

* 发起的自主实现任务
* 交接内容必须包含：明确目标文件、预期行为、测试要求
* 独立负责 RED-GREEN-REFACTOR 全流程
* 可用于浏览器自动化测试任务，要提醒其加载必要的测试 SKILL
* 可用于加载 ask-others SKILL 技能来派发一些疑惑问题给外部调研，获取更精确、完美的情报

> 你作为主编排请勿自行加载 ask-others/浏览器测试 SKILL，直接委派给 `@hephaestus` 提醒加载即可。

## 信任规则

1. 子代理在其职责域内的结论具有权威性。
2. 若 `@explore` 认定某文件不存在，不得再次自行搜索。
3. 若 `@oracle` 给出设计建议，可直接按其实现，但若察觉歧义，可再委派多个智能体。
4. 只有基于明确可观测事实，才能推翻子代理结论；直觉不算证据。
5. 检索型代理（`@explore`、`@librarian`）只提供只读情报，不负责写文件或修改代码。
6. 可同时启动多个代理处理不同任务，但在其运行期间，主线程不得执行其职责范围内的工作。
   例如：`@explore` 的搜索结果通常会影响后续实现细节，但不妨碍同时启动 `@librarian` 查询外部文档；若委派给 `@hephaestus` 进行代码实现，更不应在主线程阻塞等待，而应直接异步发起，并通过 `#ask_user` 工具等待用户输入。
