# AI Testing Workflow — 操作手册

按软件工程四阶段测试体系，指导 AI 对项目执行完整、可验证的测试覆盖。

---

## 核心心智模型

```
可测试性重构（Phase 0）
    ↓
单元测试（Phase 1）→ 每个方法都正确
    ↓
集成测试（Phase 2）→ 模块协作正确
    ↓
系统测试（Phase 3）→ 整体可用
    ↓
确认测试（Phase 3）→ 满足用户需求
```

每个阶段发现的问题类别不同，跳过任何阶段都会留下盲区。

---

## 测试四要素（贯穿所有阶段）

无论哪个阶段，每个测试用例必须包含：

| 要素 | 含义 | 缺失后果 |
|------|------|----------|
| **测试数据** | 输入值 + 边界值 + 异常值 | 只测了 happy path，遗漏边界 Bug |
| **前置条件** | 系统/环境在测试前必须处于的状态 | 测试不可重复，时过时不过 |
| **测试步骤** | 按顺序执行的操作序列 | 步骤模糊 → AI 自行发挥 → 测试无意义 |
| **预期输出** | 操作后应该观察到的结果（值 / 状态 / 副作用） | 无断言 = 没测 |

---

## 测试方法速查

| 方法 | 视角 | 适用阶段 | 设计依据 |
|------|------|----------|----------|
| **白盒** | 看得见代码内部结构 | 单元测试 | 路径覆盖 · 逻辑覆盖（判定/条件/分支） |
| **黑盒** | 只看输入输出，不看内部 | 系统测试 · 确认测试 | 等价类 · 边界值 · 场景法 |
| **灰盒** | 知道接口契约，不看实现 | 集成测试 · API 测试 | 接口 Schema + 数据库状态验证 |

## 阶段路由表

根据当前任务阶段，读取对应 reference 获取详细操作指南：

### Phase 0 — 可测试性重构（测试前必须）

**触发条件**：代码存在硬编码依赖、巨型方法、紧耦合模块等阻碍测试的问题。

**核心任务**：
1. 应用 SOLID 原则（重点：依赖倒置）使代码可注入、可 Mock
2. 探索并应用适合项目的主流设计模式，达到高内聚低耦合
3. 前后端分离项目确保 RESTful API 契约清晰

→ 当需要重构代码以支持测试时，读取 `references/testability-refactor.md`

---

### Phase 1 — 单元测试

**目标**：每个方法/函数在各种输入下行为正确。

**核心指标**：
- 覆盖率 ≥ 70%（行 + 分支）
- 白盒测试方法（判定覆盖 + 边界值）
- 全部依赖 Mock

**AI 快速指令**：
```
对 {文件} 编写单元测试，覆盖率 ≥ 70%，使用白盒方法，
外部依赖全部 Mock，每个用例包含测试数据 + 前置条件 + 步骤 + 预期输出。
```

→ 当需要编写或指导单元测试时，读取 `references/unit-testing.md`

---

### Phase 2 — 集成测试

**目标**：模块间协作正确，接口契约不断裂。

**核心指标**：
- 模块间使用真实实现，外部依赖 Mock
- 依赖倒置是 Mock 的前提（硬编码依赖 → 先回 Phase 0）
- 验证调用参数、调用顺序、状态变更

**AI 快速指令**：
```
对 {模块A} 和 {模块B} 的交互编写集成测试，
模块间用真实实现，外部依赖 Mock，验证交互行为和数据一致性。
```

→ 当需要编写或指导集成测试时，读取 `references/integration-testing.md`

---

### Phase 3 — 系统测试 + 确认测试

**目标**：整个系统在真实环境中可用，满足用户需求。

**包含三个子类型**：

| 子类型 | 内容 | 工具 |
|--------|------|------|
| API/接口测试 | 真实 HTTP 请求验证后端 | httpx / requests / supertest |
| 界面测试 | 浏览器自动化验证前端 | Playwright（首选）/ Selenium |
| E2E 管线测试 | 多阶段管线逐步验证（已内嵌完整方法） | 自定义脚本 |
| 确认测试 | 基于用户故事的验收验证 | Given-When-Then → Playwright/API |

**AI 快速指令**：
```
对系统执行完整的系统测试：
- 后端：对所有 API 端点发送真实请求，验证完整业务流程
- 前端：使用 Playwright 测试所有用户可见页面
- 每个 UI 操作必须验证所有可观察副作用（不只是被操作元素）
```

→ 当需要编写或指导系统测试/确认测试时，读取 `references/system-testing.md`（已内嵌完整 E2E 管线调试方法）

→ 当需要编写 Playwright 测试脚本时，同时加载 `playwright` Skill

## 执行顺序协议

### 完整测试流程（新项目 / 重构后）

```
Phase 0（可测试性评估 + 重构）
  → Phase 1（单元测试，覆盖 ≥ 70%）
    → Phase 2（集成测试，模块交互验证）
      → Phase 3（系统测试 + 确认测试）
```

**不可跳过规则**：
- Phase 0 发现硬编码依赖 → 必须重构后才能进 Phase 2（Mock 无法工作）
- Phase 1 覆盖率不达标 → 不进 Phase 2（基础不牢）
- Phase 2 发现接口断裂 → 修复后再进 Phase 3

### 局部测试（已有代码 / 新增功能）

可以只执行特定阶段，但必须确认前置阶段已满足：

```
只写单元测试 → Phase 1（确保依赖可 Mock）
只写集成测试 → Phase 2（确保已有单元测试兜底）
只做系统测试 → Phase 3（确保系统已部署）
```

---

## 前端 vs 后端测试差异

| 维度 | 后端 | 前端 |
|------|------|------|
| 单元测试 | 方法/函数级 | 组件渲染 + 工具函数 |
| 集成测试 | Service → Repository 交互 | Component → Store/Hook → API Service |
| 系统测试 | 真实 HTTP 请求 | Playwright 浏览器自动化 |
| Mock 重点 | 数据库、外部 API | API 调用层、定时器 |
| 特有概念 | — | 状态机测试、DOM 交互、快照测试 |
| 框架 | pytest / JUnit / go test | vitest / jest / @testing-library |

---

## 覆盖率指标

| 阶段 | 覆盖率目标 | 衡量方式 |
|------|-----------|----------|
| 单元测试 | ≥ 70% 行 + 分支 | 覆盖率工具报告（istanbul / coverage.py） |
| 集成测试 | 关键交互路径 100% | 所有模块间接口都被调用过 |
| 系统测试 | 所有用户可见功能 | 功能清单逐项打钩 |
| 确认测试 | 所有验收标准 | 用户故事逐条验证 |

---

## 禁止事项

- **禁止**：没有前置条件的测试用例（不可重复 = 无意义）
- **禁止**：没有断言的测试（只调用不验证 = 没测）
- **禁止**：Mock 具体实现而非接口（脆弱，实现一改就挂）
- **禁止**：跳过 Phase 0 直接写集成测试（硬编码依赖 Mock 不了）
- **禁止**：UI 测试只验证被操作元素（必须验证所有副作用）
- **禁止**：删除失败的测试来"通过"
- **禁止**：在测试中硬编码 sleep 等待（用显式等待条件）
- **禁止**：测试之间有顺序依赖或状态泄漏
