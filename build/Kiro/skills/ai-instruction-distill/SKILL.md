# AI Instruction Distill

把素材蒸馏为面向 AI 的最优 token 指令文件。本 skill 只管"怎么产出"，抽象模型由 `context-engineering.instructions.md` 承载，通用工程原则由代码规范类 instructions 承载。调用不复制。

---

## 蒸馏流程

### 1. 识别消费者

回答三个问题：

- 这份产出会被谁加载？（全局 AI / 项目 AI / sub-agent / 特定 skill 触发）
- 在什么场景下加载？
- 消费者能访问哪些资源？（可见性模型：消费者上下文 = 你交付的 + 他可达的）

**输出**：一行消费者声明。若消费者不是 AI（比如是人类读者），停止——本 skill 不适用。

### 2. 两层蒸馏

**第一层 · 抽象到问题本质**

从素材提取，不复述素材：

- 素材针对的**行为问题**是什么？（不是它描述的现象，是现象背后的可操作缺陷）
- 它用什么**结构 / 判据 / 约束**解决？
- 剥离原主题，编码为跨场景规则。丢弃原主题词、案例名、原句。

**第二层 · 按消费者分流**

既然下游是 AI：只保留问题本质 + 可执行判据。删除情境叙事、现象描写、共鸣段落。

**自检 · 替换测试**：把产出中所有具体主题词换成不相关主题，规则仍成立 → 到位；不成立 → 停在实例层，回 Step 2 重做。

### 3. 起草规则

形态约束：

- 主体用条件句 / 动作句 / 边界句
- 作用域显式声明：适用场景、不适用场景、消费者类型
- **写判据不写模板**：规则定义什么算对，不锁死措辞 / 字数 / 结构
- 预判退化输入（极短 / 极长 / 信息不足 / 自相矛盾），给判据不给固定答案
- 自包含或可追溯二选一：关键语义当前文件内完整，或路径 + 行号 + 相关性说明

### 4. 跨文件 DRY 检查

列出同级 instructions 目录下所有现有文件，逐一排查内容重叠：

| 已由他处承载 | 本文件处理方式 |
|------------|-------------|
| 通用工程原则（DRY/KISS/YAGNI/First Principles/原子化/标准化命名） | 不写，调用 code-quality instructions |
| 消费者模型、可见性、抽象层级、信息密度、判据 vs 模板 | 不写，调用 context-engineering instructions |
| 具体语言规范（TypeScript/Python/Java 等） | 不写，调用对应语言 instructions |
| 其他已有 instruction 的具体规则 | 不复述，引用路径 |

原则：调用不复制。

### 5. 自我审计（5 条扫描清单）

每次扫描全文，命中即修复：

**扫描 A · 元叙事残留**
- 对委托者回应类："我理解了…"、"根据你的需求…"、"按照您的要求…"
- 自我评价类："这样设计的优势是…"、"本方案整体…"
- 结构汇报类："以下将介绍…"、"综上所述…"
- 状态标记类：正文里嵌入 "(已简化)"、"(已更新)"、"(完整版)"、"(修订后)"
- 命中处理：删除或重写为事实陈述

**扫描 B · 装饰性自指节**
- "本文件只写…"段 → 与 frontmatter description 重复
- "## 递归约束"节 → 文档自我描述，对下游行为无增量约束
- "## 作用域"节若复述 description → 删除
- 命中处理：删除；description 是唯一定义位

**扫描 C · 展开解释句**
- 规则主句后的"因为…"、"这样做是为了…"、"其目的在于…"
- AI 消费者不需要被说服理解规则，规则本身可执行即可
- 命中处理：删除解释部分，保留规则主句

**扫描 D · 跨节同义重复**
- 同一原则在不同章节用不同措辞写过 ≥2 次
- 常见：信息密度 / 可见性 / 判据 vs 模板 散落多处
- 命中处理：合并到最权威的一节，其他位置最多一句调用

**扫描 E · 层级混淆**
- 抽象层文件混入具体示例、触发词、反模式词库
- 应用层文件混入通用工程原则
- 命中处理：示例下沉到应用层或独立 SKILL；通用原则上沉到根规范

### 6. 三遍砍法收敛

第一遍：按 Step 5 清单砍
第二遍：每句问"删掉后消费者的执行 / 理解 / 判断会变吗？" 不变 → 砍
第三遍：每节问"这节与其他节有重叠吗？" 有 → 合并

停止条件：再砍即损伤信息保真。

## Gotchas（真实踩坑录）

每条标注识别方法与修正方式。

**Gotcha 1 · 顶部前言复述 description**
识别：文件开头紧跟 heading 的段落是否在解释"本文件承担什么"？
修正：description 已说；前言段删除。

**Gotcha 2 · 写"递归约束"节**
识别：是否有一节在讲"本文档自身也受自身约束"？
修正：删除。任何规则文件被 AI 加载后都会被应用到所有相应产出，显式声明自指是装饰。

**Gotcha 3 · 为"负责任"加解释句**
识别：规则条目后是否有"因为…"、"会导致…"、"这样做能…"的铺垫？
模型根因：LLM 训练偏好"人读顺畅 = 好"，潜意识把解释当负责。但 AI 消费者不需要被说服。
修正：规则陈述完即止。示例和解释只在"无此即执行歧义"时才加。

**Gotcha 4 · 用硬模板替代判据**
识别：规则是否锁定了措辞、字数、章节顺序、句式？
典型错例："必须按以下 7 层顺序写：..."、"每条规则必须以动词开头"
修正：改为"满足 X 判据 / 不出现 Y 行为 / 可由 Z 验证"。例外：schema、机器契约、commit / 日志强一致格式。

**Gotcha 5 · 停在现象复述层**
识别：替换测试失败——换个主题规则就不成立。
典型错例：写"AI 妄想螺旋"规则时，规则里出现"ChatGPT"、"妄想螺旋"等原主题词。
修正：回 Step 2 第一层，重新识别行为问题本质，剥离原主题。

**Gotcha 6 · 悬空引用**
识别：是否出现"参考上文"、"根据之前的结论"、"如前所述"、"按某模板"？
典型错例：委托 sub-agent 时写"根据刚才的结果修复…"
修正：改为自包含（关键语义当前 prompt 内完整）或可追溯（路径 + 行号 + 相关性）。

**Gotcha 7 · 状态标记泄漏到正文**
识别：产物正文里出现 "(已简化)"、"(已更新)"、"(完整版)"、"(修订后)"。
模型根因：模型把"对委托者的进度汇报"错放进了"产物本体"。
修正：这些信息属于对话层，不属于产物层；产物里删除。

**Gotcha 8 · 跨文件重复通用原则**
识别：你的新 instruction 里写了 DRY / KISS / YAGNI / First Principles / 原子化。
修正：这些已在 code-quality 或根 AGENTS.md 承载。改为一句"适用通用工程原则（见 code-quality instructions）"，不复述。

## 验收判据

产出可交付的硬条件（全通过才算完成）：

- ✅ 替换测试通过：规则换主题仍成立
- ✅ 5 条扫描清单全部清空
- ✅ 作用域显式声明（适用 / 不适用 / 消费者类型）
- ✅ 三遍砍法已执行
- ✅ 无跨文件重复：列出依赖的其他 instruction，确认只调用不复述
- ✅ 信息保真：与初稿对比，所有原始规则、判据、语义仍然可从终稿恢复
