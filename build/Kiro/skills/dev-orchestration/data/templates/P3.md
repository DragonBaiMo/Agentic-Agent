## Step: oracle-review-classify
> Agent: 编排者自身 | Phase 3 — 审核分流决策

### 执行步骤

1. **分类**（按信息来源需求归入四类）：
   - INTERNAL（内部矛盾/笔误）→ 委托 @hephaestus 修复
   - LOCAL_VERIFY（假设未验证）→ 委托 @explore 确认 → 追加修复
   - EXTERNAL_DOC（外部知识可能过时）→ 委托 @librarian 查文档 → 追加修复
   - HUMAN_RESEARCH（自动手段穷尽、需外部补充研究）→ 委托 @librarian 加载 ask-others 生成调研请求；仅在外部阻塞成立时 ask_user 呈递
2. **并行派发**：同章节合并同一 agent；不同章节可并行；BLOCKER 优先，MINOR 记录不阻塞；HUMAN_RESEARCH 等待期间其他类别继续
3. **验证**：修复完成后委托 Oracle 快速审查修复部分，确认无新 BLOCKER
4. **循环**：全部修复完成 → `dev-orch redo P3 hephaestus-design oracle-design-review oracle-review-classify` → `dev-orch dispatch P3`
5. **退出判定**：
   - 正常：0 BLOCKER + 评级 ≥ B → 进入下一阶段
   - 强制：循环 ≥ 3 轮仍有 BLOCKER → 编排者决策
   - HUMAN 等待：其他类别已清零 → 先进 P4


### 审核记录（每轮必做）
每轮追加 `{{LOG_DIR}}/review-log.md`（轮次/评级/发现数/分类分布/未解决 BLOCKER）。BLOCKER/RISK 出现时创建 `{{LOG_DIR}}/blocker-tracker.md`（ID/级别/描述/来源/负责/状态/解决方案），已解决标 ✅。

**完成验证** 所有 BLOCKER 已清零或走强制退出 / 评级 ≥ B / `{{LOG_DIR}}/review-log.md` 已追加本轮记录

**退出**
```

dev-orch mark P3 oracle_rating_B_plus passed "{{REVIEW_DIR}}/P3-oracle_rating_B_plus.md"
若命中三类 ask_user 例外，再在收到用户结论后执行：
```
