# SubAgent Instruction Pack

## 适用边界
1. 仅用于实现类委托：@hephaestus、@explore-写入、@librarian-深度研究。
2. 不用于纯咨询类委托：@oracle、@metis、@librarian（只读研究）。
3. 不用于编排者自身步骤；仅作为发布给 SubAgent 的 prompt 预制段。
4. 不改变业务目标，只约束代码注释与标签纪律。
5. 不可删改原任务的 MUST DO/MUST NOT DO，仅可追加本规范。

## 注释规范
1. 仅在复杂逻辑前写简短注释。
2. 注释必须解释 why，不复述代码表面行为。
3. 禁止显而易见注释，例如“赋值变量”“循环数组”。
4. 禁止用注释掩盖糟糕命名；优先改名。
5. 注释长度控制在 1-2 行，保持可扫描。
6. 注释必须与代码同步；代码变更时同步更新注释。
7. 注释失真视为缺陷，必须在本次改动中修复。
8. 禁止堆叠历史说明；历史记录写入提交信息或日志文件。
9. 禁止空泛词汇：优化、处理、兼容、增强（无上下文）。
10. 公共 API 可写简短 docstring；内部私有逻辑仅在复杂处注释。

## TODO/FIXME/HACK/NOTE 标签规范
1. 仅在确有未完成事项时使用标签。
2. 统一格式：`TODO(owner,date): ...`。
3. 统一格式：`FIXME(owner,date): ...`。
4. 可选格式：`HACK(owner,date): ...`。
5. 可选格式：`NOTE(owner,date): ...`。
6. owner 必填，使用稳定身份（如 hephaestus、alice）。
7. date 必填，使用 `YYYY-MM-DD`。
8. 描述必须可执行，包含明确后续动作。
9. 禁止泛化 TODO，例如“后续优化”“以后再改”。
10. 禁止无 owner/date 的裸标签。
11. 一个未完成事项对应一个标签，避免一条覆盖多问题。
12. 问题已完成时必须删除标签，禁止遗留过期标记。
13. 标签用于技术债记录，不用于记录普通解释。
14. 涉及风险时优先使用 FIXME/HACK，普通待办使用 TODO。

## 输出要求
1. SubAgent 最终汇报必须列出新增/修改的标签位置。
2. 位置格式：`<path>:<line> <tag>`。
3. 必须说明每个标签的原因与后续处理条件。
4. 若本次无标签变更，明确写“无新增/修改标签”。
5. 若修改了注释，汇报中列出关键注释所在文件。

## 最小注入片段模板（复制到 7-segment MUST DO）
```md
- 发布前强制注入并遵循 `{{INSTRUCTION_PACK_PATH}}`。
- 注释规则：仅在复杂逻辑前写 1-2 行 why 注释；禁止显而易见注释；注释必须与代码同步。
- 标签规则：仅在确有未完成事项时使用 `TODO(owner,date): ...` / `FIXME(owner,date): ...`；禁止泛化 TODO。
- 最终汇报必须列出本次新增/修改标签位置：`<path>:<line> <tag>`。
```

## 执行检查清单
1. 发布委托前，确认 prompt 的 MUST DO 已包含注入片段。
2. 回收结果时，检查是否出现裸 TODO/FIXME。
3. 回收结果时，检查标签是否满足 owner/date 格式。
4. 回收结果时，检查复杂逻辑是否有必要注释。
5. 不满足任一项时，驳回并要求 SubAgent 修正后重提。
