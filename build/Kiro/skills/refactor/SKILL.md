# 重构工作流

## 前置：一句话目标

用一句话定义重构目标。说不出来就先和用户澄清。

> 示例："将 `config-manager.ts` 中的校验逻辑提取为独立的 `config-validator.ts`，使原文件降到 300 行以下。"

这句话是范围契约。**超出范围的一切不做**。

---

## Phase 1 — 分析

1. 完整读取目标文件
2. 找出所有引用方（谁导入/调用了目标中的符号）
3. 建立依赖图：

示例：
```
| 文件 | 角色 | 引用方向 |
|------|------|----------|
| `target.ts` | 目标 | — |
| `caller-a.ts` | 调用方 | 导入目标 |
| `shared/index.ts` | 桶文件 | 重新导出目标 |
```
**阻塞条件**：发现循环依赖 → 先标记再商议方案，不直接开拆。

---

## Phase 2 — 安全：定位测试覆盖

确认即将移动的代码有测试守护：
示例：
```
| 情况 | 操作 |
|------|------|
| 测试存在，覆盖待移动符号 | 继续 |
| 测试存在，但不覆盖待移动符号 | 先补测试再重构 |
| 无测试 | 至少写一个锚定公共 API 的集成测试 |
```
**铁律**：不移动无测试的导出函数。测试不必穷举，但须证明契约存在。

---

## Phase 3 — 范围定义

明确边界清单：

- [ ] 列出所有要移动的符号（逐个点名）
- [ ] 列出所有留在原地的符号
- [ ] 列出新文件需要的导入
- [ ] 列出现有调用方需要更新的导入
- [ ] 列出需要更新的桶文件/模块入口

**范围蔓延规则**：不在「移动」清单上的符号，即使「顺手更干净」也不动。范围变更须重新定义目标句。

---

## Phase 4 — 增量执行

一次只移一组概念。禁止在同一次编辑中移动多个不相关分组。

### 执行顺序（依赖安全）

1. 创建目标文件，先写类型导入（无业务逻辑）
2. 从最低依赖的符号开始移动（纯函数 → 方法 → 含状态组件）
3. 更新模块入口/桶文件的导出
4. 逐个更新调用方，每个确认编译后再下一个
5. 最后从源文件删除已移动的符号

### 执行约束

- 文件命名遵循项目约定（通常 kebab-case）
- 新文件须有单一内聚职责（一个名词短语可概括）
- 禁止创建 `utils.ts` / `helpers.ts` / `common.ts` 等笼统文件
- 新文件中至少有 2 个导出符号（单符号模块通常是错误信号）
- 禁止引入 `any` 类型抑制或忽略注解

---

## 重构模式

### 提取函数

**时机**：函数体超 ~30 行，或包含两个以上不同关注点。

**信号**：代码中用注释块分隔不同段落 → 每段是候选提取点。

**规则**：
- 提取后的函数先放在同文件内
- 仅当被多个调用方使用时才提取到独立文件
- 提取后原函数应读起来像伪代码（明确的步骤编排）

### 提取模块

**时机**：文件超过项目行数限制，或包含可独立删除的不相关函数。

**前置检查**：
- [ ] 新文件职责可用一个名词短语描述
- [ ] 新文件中无符号需要从旧文件反向导入（避免循环依赖）
- [ ] 至少 2 个导出符号

### 重命名

**时机**：名称误导、使用缩写、与上下文术语不匹配。

**命名约定参考**：

| 符号类型 | 约定 | 示例 |
|----------|------|------|
| 布尔变量/属性 | `isXXX` · `hasXXX` · `canXXX` | `isDisabled` · `hasItems` |
| 事件处理 | `onXXX` · `handleXXX` | `onSessionCreated` |
| 类型/接口 | PascalCase 名词 | `AgentConfig` · `UserState` |
| 模块级常量 | SCREAMING_SNAKE | `DEFAULT_TIMEOUT_MS` |
| 文件 | kebab-case | `agent-builder.ts` |

**优先使用 IDE 的重命名功能**（原子性更新所有引用），避免手动查找替换。

### 降低耦合

**时机**：模块直接导入并调用具体依赖（可注入替代）；或两个模块互相导入（循环依赖）。

**依赖注入优于直接导入**：
- 抽取接口/类型描述依赖行为
- 调用方通过参数传入实现
- 便于测试替换

**循环依赖修复**：A↔B 互相导入 → 提取共享类型/常量到第三方文件 C，A 和 B 各自只从 C 导入。

---

## Phase 5 — 验证

每移动一组符号后（不是最后才验证）：

1. 静态检查（typecheck / lint）→ 零新增错误
2. 运行受影响路径的测试 → 全过
3. 构建（如适用）→ 退出码 0

**验收标准**：
- [ ] 定检查零新增诊断
- [ ] 所有既有测试通过
- [ ] Phase 2 新增的测试通过
- [ ] 无类型抑制或忽略注解引入

---

## Phase 6 — 清理

验证通过后：

1. **死导入**：删除源文件中不再使用的 import
2. **死导出**：审查是否有不再需要的显式重导出
3. **过时注释**：删除描述重构前状态的注释和 TODO
4. **最终验证**：再跑一次静态检查 + 测试

---

## YAGNI 守卫

重构期间拒绝以下诱惑：

| 诱惑 | 为什么错 |
|------|----------|
| 「顺手加个接口以备扩展」 | 没人要求；死抽象 |
| 「泛化一下万一别处用到」 | 别处还没用到 |
| 「顺手加个错误处理」 | 范围蔓延；另开任务 |
| 「把旁边不相关函数也重构了」 | 需要新的目标句 |

**重构在原始目标句满足时结束。仅此而已。**
