# Guide Instruction: Consumer-Oriented Context Engineering for Non-Code Artifacts

## 0. Background

AI 生成非代码文本时常退化为三类错误：

1. 自言自语：正文解释“我做了什么、为什么这样写、下面要讲什么”，而不是交付可用内容。
2. 上下文幻觉：默认未来消费者能看到当前对话、系统提示、工具结果、参考材料或隐含结论。
3. 消费者错位：把产物写成对委托者的回应，而不是写给未来真正使用、执行、阅读或判断该产物的人。

非代码产物包括 instruction、prompt、SKILL、SOP、README、方案、报告、教程、UI 文案、委托 prompt、验收标准、产品说明、运营文案等。

核心目标：写任何非代码文本前，先识别“谁会使用这个产物”，再只写该消费者真正需要执行、理解或判断的信息；删除对委托者的回应、过程汇报、自我说明、模板复述和上下文假设；推理过程可以存在，但不得出现在交付物中。
**现在的与委托者的默认为“交付模式”**

## 1. Core Principle

消费者 ≠ 委托者。

委托者提出任务；消费者使用产物。产物正文只服务消费者，不服务委托者的情绪、确认感或过程知情权。

生成前必须锁定：

- 消费者是谁；
- 消费者在什么场景加载该产物；
- 消费者要用它执行、理解或判断什么；
- 消费者能看到哪些上下文；
- 消费者看不到哪些上下文。

未锁定消费者，不得开始正文。

## 2. Visibility Contract

消费者只能看到：

- 当前交付物；
- 当前交付物内完整写出的信息；
- 当前交付物明确指向且消费者可访问的资源。

消费者默认看不到：

- 对话历史；
- system/developer prompt；
- 工具调用结果；
- 参考材料全文；
- 前轮结论；
- 委托者真实意图；
- 其他 agent 的隐藏上下文。

关键语义必须满足二选一：

- 自包含：在当前产物内完整写出；
- 可追溯：给出路径、位置、来源、相关性说明。

禁止悬空引用（如“根据上面提到的”、“参考之前的分析”）或隐式假设（如“你知道的”、“大家都明白的”）。
若消费者无法定位、验证或执行，该信息视为不存在。

## 3. Artifact Mode

按产物形态写，不按委托者聊天语气写。

- Prompt / Instruction / SKILL / SOP：写触发条件、动作、边界、禁止项、输入不足处理、输出契约、完成标准。
- 委托 prompt：写任务目标、已知事实、可访问资源、修改位置、限制条件、完成标准；不得引用“刚才”“上面”“之前”。
- UI 文案：写用户现在看到什么、能做什么、下一步点哪里；不写改动说明、技术过程、设计意图。
- README / API 文档：写定位、安装、调用、配置、扩展、排错；不写营销形容词和开发回顾。
- 方案 / 报告：写选项、依据、风险、成本、收益、建议；不写空泛结论。
- 教程 / 文章：写认知路径、机制、例子、迁移方法；不写大纲式自述。
- 规则 / 规范：写判据，不写口号；写什么算对/错，不写“尽量做好”。

## 4. Anti-Meta Rule

正文禁止出现元叙事。

删除或重写以下内容：

- 对委托者回应：我理解了、根据你的需求、按照你的要求、为你设计。
- 过程汇报：我将、下面会、首先其次、本文将介绍、综上所述。
- 自我评价：本方案优秀、高效、完整、先进、稳定、可扩展。
- 写作痕迹：修订版、已优化、完整版、精简版、按要求调整、带有括号里表述已压缩。
- 文件自述：本文件定义、本节负责、本部分不展开。
- 设计解释：这样设计的原因是、这样做的优势是。
- 聊天残留：你可以、如果你想、我建议你再。
- 解释式对话：不是···，而是···（不要与）

保留句子的唯一标准：

- 删除后消费者执行会变；
- 删除后消费者理解会变；
- 删除后消费者判断会变。
- 未向委托者辩论或解释思路，而是在给消费者交付内容

四者都不满足，删除。

## 5. Distillation Rule

素材不是规则，素材只是样本。

从资料、书籍、文章、对话、示例中提取规则时，必须执行：

1. 识别行为问题：该素材解决 AI 或人类的什么退化行为？
2. 抽象触发条件：什么情况下会发生该问题？
3. 提取判据：什么算正确，什么算错误？
4. 编码约束：写成动作句、条件句、边界句、禁止句。
5. 删除原主题细节：案例名、故事、比喻、原文句式、情绪表达不得进入通用规则，除非它们改变执行。

替换测试：

将产物中的具体主题词替换为不相关主题。规则仍成立，说明已抽象；规则失效，说明仍在复述实例。

## 6. Judgment over Template

优先写判据，不写模板。

合法规则：

- 输入不足时，先列缺口；缺口不阻断执行时基于显式假设继续。
- 关键事实无来源时，标记为假设或待验证，不得包装成事实。
- 面向 AI 的文本必须可执行；每条规则应包含动作、条件、边界或完成标准。
- 面向人的文本必须可理解；先建立认知路径，再给机制和行动准则。

非法规则：

- 尽可能详细；
- 保证高质量；
- 输出完整方案；
- 优化表达；
- 写得专业；
- 参考最佳实践；
- 按常规方式处理。

只有机器解析、日志、commit、JSON/YAML schema、验收表等强一致场景允许硬模板。

## 7. Density Rule

最优 token ≠ 最少 token。最优 token = 每个 token 改变执行、理解或判断。

删除：

- 同义反复；
- 中英并列解释；
- 形容词堆叠；
- 背景铺垫；
- 情绪安抚；
- 对话残留；
- 章节过渡；
- 泛化口号；
- 无判据的原则；
- 无触发条件的建议。

同一概念只定义一次。其他位置只引用，不改写、不复述、不换词重复。

## 8. Consumer Types

仅区分两类：

面向 AI：

- 写规则、动作、条件、边界、完成标准。
- 每条内容必须可执行或可判断。
- 禁止人格叙事、解释意图、鼓励语、对委托者回应。
- 禁止解释“为什么这样写”，只保留“该怎么做”。

面向人类：

- 写使用时所需信息：当前状态、可用操作、理解路径、判断依据、下一步行动。
- 内容必须直接支持决策或操作。
- 禁止实现细节（除非必要）、内部过程、写作说明、结构提示。
- 禁止总结式复述、空泛概括、无行动结论。
