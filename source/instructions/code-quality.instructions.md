**编码前先思考**：不要想当然，不要掩盖疑惑，主动暴露权衡关系。真正开始实现前，先问问自己：你现在的做法，真的符合、遵守你的 `SOUL` 吗？

# 代码规范

## Style

- Google Style Guides 中对应语言的风格规范：(有实际意义的)命名、大小写、格式化、注释、文件结构与组织方式。
- Google Engineering Practices 与 Google’s Code Review Guidelines 中指南：确保代码具备高可读性、高可维护性、高可测试性、低复杂度、长期可演进性，并符合工程化协作与代码评审标准。
- 遵循 SOLID 原则，Every File 软上限 200 行；>500 行时必须想办法拆分、重构或引入新文件
- import 通过模块公共入口，不直接引用内部文件；禁止循环依赖

```???
**计划**：互斥子任务拆分 · 列出改/增/不改文件 · 大任务计划写入文件防截断。

**实现**：。
  - 方法 30-200 行，文件 >500 行拆分。
  - 同一文件可直接修改、重构或合并输出。
```

## Safety

- 默认采用 Secure by Design、Secure by Default、Defensive Programming 与 Design by Contract 思维。
- 实现前：先识别信任边界、安全不变量、敏感数据、权限要求和误用场景。
- 实现时：默认所有外部输入不可信；在边界处校验，对所有敏感操作执行授权检查；使用最小权限减少暴露面，避免泄露密钥和内部细节；失败时默认拒绝；
- 实现后：在必要情况为关键业务的 边界/滥用 路径补充安全测试。
- 禁止任何手段抑制或绕过代码检查

## 注释（代码变更时必须同步更新）

**注释与代码矛盾比无注释更糟**

### 可用标签
- `NOTE: [说明]` → 已采纳的设计决策/非显而易见的业务规则/重要提醒/额外说明
- `FIXME` → 代码有问题/不能正常工作/有潜在风险需尽快修复
- `TODO(owner): [描述]` → 待办功能/未来优化或重构（非紧急）/暂未完成/等待获取信息/遇Blocked
- `PLACEHOLDER: [解释]` → 实现虽可工作但有待商榷/临时方案/占位实现/Mock/拿不准/需进一步思考

> 其它如：REVIEW（需评审）、WARNING（潜在风险）、OPTIMIZE（优化点）

### 风格
- 写 **why**
- 简洁明了，避免行话；禁止套话和描述式注释
- 勿描述显而易见代码
- 注释写给“下一位贡献者”（可能是自己），慷慨提供必要信息，提供上下文
- 风格一致：项目/团队统一位置、格式、标点
- 禁止重复在代码中显而易见的已有信息
- 禁止完整句式和冗余措辞

```ts
// ❌ 禁止 — 重复代码已表达的信息
// Set timeout to 30 seconds
const TIMEOUT = 30_000

// ✓ 正确 — 每个公共方法必有注释
// [···]
public isValid(input: unknown): boolean { ... }

// ✓ 正确 — 解释 why
// NOTE: 上游 API p99 ≈ 25s，留 5s 余量
const TIMEOUT = 30_000

// ✓ 正确 — 精简
// 支付网关偶发瞬时错误 → 重试 3 次，指数退避
```

### 必写注释的情况：
1. 文件/模块级：许可声明 + 简述文件用途、内容、用法
2. 类/结构：描述用途、如何使用、同步/线程假设、关键不变式；可附简单示例
3. 每个公共方法/函数：声明处说明输入/输出、边界条件、性能影响、非null要求；定义处说明实现技巧/步骤/选择原因（不重复声明注释）
4. 关键变量：非显而易见常量/哨兵值含义、约束
5. 实现细节：棘手/重要代码块前 需解释

## Log

- **什么情况下写**：关键事件、潜在问题、错误细节、主业务流程
- **不写什么**：不记录一切（成功200/300状态码通常省略）、不记录源代码/完整SQL/未脱敏数据

### 日志级别

按等级从低到高进行忽略显示，如设置为1级则无法显示高于该等级的日志：
- lv0：
  - NONE：不记录任何日志
- lv1：
  - CRITICAL/FATAL：严重错误（服务崩溃、不可恢复故障）
  - ERROR：影响功能的错误/异常（数据库失败、业务逻辑错误）
  - WARN：潜在问题/轻微异常（超时、缓存接近满、弃用API）
  - SECURITY：安全相关事件（认证失败、权限变更、敏感操作等）
  - INFO：重要业务/系统事件（启动、关键流程、用户操作、配置加载）
- lv3：
  - DEBUG：细粒度调试信息（所有细节）

#### 常见错误

- 不用 `print()` 代替 logger（缺少元数据、无法分级）
- 不在 循环/高频路径 无条件记录
- 库代码仅添加 `NullHandler`，不配置具体 Handler

#### 规范

- **结构化日志优先**：JSON 格式（含 timestamp、level、message、trace_id 等字段），需便于 ELK/Loki/Splunk 等工具搜索、聚合、告警；避免纯文本拼接。
- **性能与安全**：使用异步日志、采样（高流量场景）；永不记录敏感数据（密码、PII、密钥、银行卡等），需脱敏/哈希/加密；日志变更时同步更新。
- **统一**：定义固定 schema（字段名一致），每个模块使用专用 logger。
- **明细**：可随时调整等级，用于在开发环境详细、生产环境精简。


## 临时产物管理

统一在项目目录 `tmp/` 下，严禁散落到业务代码目录下

涵盖：
- 临时脚本、调试、一次性任务、数据修复、SQL 执行（及其对应生成物如报表、导出数据、截图）
- 仅临时参考的外部资源
- 从外部拷贝的代码片段或信息快照
- 调试日志 / 抓包 / profile 文件

规则：
- `tmp/` 必须加入 `.gitignore`，禁止提交
- 命名：需在`tmp/`目录下按用途划分子目录，如 `{用途}-{YYYYMMDD}.{扩展名}`，禁止任何无语义命名或直接放在`tmp/`直系目录下
- 留存：用完可不删（保留上下文），但禁止移出 `tmp/` 混入业务目录
- 临时产物可转为正式产物，但必须按照编码风格重构命名、注释、日志等，再移入目标业务代码

### 其它

`.bat` / `.ps1` 脚本执行或编码出问题 → 优先考虑 Nodejs / Python 包装








# 代码质量规范

交付的代码必须：可运行 · 可验证 · 可维护。

---

## 核心原则

1. **围绕验收标准**：不做范围外优化
2. **先结构后代码**：先定模块 / 接口 / 数据流
3. **规划驱动**：非琐碎变更先给计划
4. **不确定即停**：不猜 API / 包名 / 参数；未指定的第三方须查验公开文档
5. **一次一模块**：单变更闭环再进下一模块
6. **接口先行**：契约定义先于实现
7. **上下文管理**：代码量大时切会话，关键决策落文件
8. **最小变更**：默认 Minimal Diff；重构不改业务逻辑（除非明确授权）

---

## 任务评估

接手任务前完成识别，结果决定后续力度。

### 规模 · 复杂度 · 风险

| 维度 | 小型 | 中型 | 大型 |
|------|------|------|------|
| 文件数 | ≤5 | 6-20 | >20 |
| 模块/包 | 单模块 | 2-4 | >4 |
| 外部依赖 | ≤2 | 3-8 | >8 |
| 协作方 | 仅自己 | 有交接方 | 多团队 |

| 复杂度信号 | 低 | 中 | 高 |
|------------|-----|-----|-----|
| 状态管理 | 无/单源 | 多源独立 | 跨模块共享 |
| 并发 | 同步 | 局部异步 | 全链路异步/多线程 |
| 外部集成 | 无 | 1-2 API | 多系统 |
| 数据流 | 单向 | 有回路 | 多方双向 |

| 风险信号 | 应对 |
|----------|------|
| 不可逆操作 | 确认机制 + 回滚/补偿 |
| 性能敏感 | 基准测试，不凭直觉 |
| 安全敏感 | 安全审查，遵守质量闸门 |
| 首次接触的技术栈 | 先验证 API 行为 |
| 遗留代码 | 先摸清再动，不盲从旧模式 |

### 行为适配

| 规模 | 计划 | 验证 | 文档 |
|------|------|------|------|
| 小型 | 可省略书面计划 | 静态检查+测试+回读 | 代码内注释 |
| 中型 | 书面文件清单 | 全部 4 步 | 接口注释+决策记录 |
| 大型 | 分批计划（每批可回滚） | 4 步+架构审查 | 接口文档+变更记录+交接说明 |

| 任务类型 | 额外要求 |
|----------|----------|
| Bug 修复 | 最小修复+回归测试，不顺便重构 |
| 新功能 | 接口先行，无测试不交付 |
| 重构 | 先有测试覆盖 |
| 性能优化 | 先基准后对比 |
| 模块交接 | 契约先行+边界测试+状态隔离；交付：职责描述/API签名/TODO/命令/依赖图 |

---


### 禁止模式

| 禁止 | 替代 |
|------|------|
| 空 catch / 吞异常 | 日志带上下文 + 重抛或处理 |
| 笼统文件名 | 按领域命名 |
| 硬编码配置（密钥/URL/阈值） | env / 配置注入 |
| 生产代码调试输出 | 结构化日志框架 |
| 反射 / 动态拼接绕过校验 | 正常 API 调用 |
| YAGNI 违反（提前设计） | 只做当前所需 |
| 删除失败测试 | 修复根因 |
| 散弹枪调试 | 诊断根因后最小修复 |
| 抑制/绕过类型检查 | 修复类型定义本身 |
| 裸 TODO / 无归属标记 | 带 owner + issue 编号 |

## 接口契约

### 统一响应结构

```json
{ "code": 0, "data": { ... }, "message": "操作成功" }
{ "code": 40001, "error": "VALIDATION_ERROR", "message": "用户名不能为空", "details": [...] }
```

### 状态码

| 码 | 语义 | 场景 |
|----|------|------|
| 200 | 成功 | GET/PUT/PATCH |
| 201 | 已创建 | POST |
| 204 | 无内容 | DELETE |
| 400 | 请求错误 | 参数校验失败 |
| 401 | 未认证 | 缺少/无效 token |
| 403 | 无权限 | 认证通过但无权 |
| 404 | 不存在 | 资源未找到 |
| 422 | 无法处理 | 业务校验失败 |
| 429 | 过多请求 | 限流 |
| 500 | 服务器错误 | 未捕获异常 |

### 设计要求

- 资源复数名词 · RESTful 动词 · GET/PUT/DELETE 幂等 · POST 支持 `Idempotency-Key`
- 分页：`GET /resource?page=1&size=20&sort=field:desc`
- 错误响应结构统一，禁止裸字符串
- 版本策略明确（URL `/v1/` 或 Header）
- 契约须含：字段约束 · 错误码 · 示例 payload

---

## 执行闭环

### 1. 契约 → 2. 计划 → 3. 实现 → 4. 验证 → 5. 交付

**契约**：输入/输出 · 边界 · 失败语义 · 幂等性 · 安全边界。不明确 → 查证或提问。

**验证（按序，不可跳过）**：

1. **静态检查**：typecheck / lint / format 零新增错误
2. **针对性测试**：最窄范围优先；跨模块时全量
3. **回读**：重读修改文件，确认意图 · 无调试残留 · 无副作用
4. **回归**：共享模块变更 → 构建退出码 0 → 引用方测试通过

| 触发 | 最低要求 |
|------|----------|
| 文件编辑后 | 1+2 |
| 标 TODO 完成前 | 1-3 |
| 报告完成前 | 1-4 |
| 跨模块重构 | 1-4+架构审查 |

小型项目可省步骤 4，步骤 1-3 不可跳。

**测试铁律**：
- 每个公共方法 ≥1 测试 · 每个业务流程：正常+≥2 异常
- 禁止手动验证替代 · 禁止 `TODO: 补测试` 交付 · 测试失败=交付失败
- 分层：单元（纯逻辑）· 集成（服务/DB/外部）· 回归（Bug 修复必带）
- Mock：外部依赖必须 Mock，行为须与真实语义一致

**交付检查清单**：契约 · 计划 · 改动(changeset) · 关键代码 · 命令 · 验证步骤 · 阻断项 TODO

## 质量闸门

### 安全

- 外部输入全不可信：白名单校验 · 类型/范围/长度/格式
- 机密禁止硬编码；env 注入；日志脱敏
- 防护：注入（SQL/NoSQL/命令/模板）· 路径遍历 · XSS/CSRF · SSRF · DoS
- 最小权限；敏感动作须鉴权；CORS 禁止 `*`

### 错误处理与容错

- 错误须：可定位 · 可归因 · 标注可恢复/不可恢复
- 空 catch 禁止 → 日志带上下文或重抛
- 重试：指数退避，区分可重试/不可重试
- 熔断：连续失败达阈值 → 快速失败 → 定期探测恢复
- 降级：核心优先，非核心可降级，降级状态可观测
- 幂等：创建前检查存在性，破坏性操作须确认
- 可恢复：持久化操作须有回滚/补偿机制

### 性能与并发

- IO/网络/重计算优先异步，标明线程安全边界
- backpressure · 超时 · 重试 · 熔断
- 共享状态加锁/原子操作；避免 TOCTOU
- 关注算法复杂度

### 数据与资源

- 参数化查询 · 事务边界明确 · 迁移可回滚
- 资源显式释放（`try-finally` / `using` / `with`）
- 缓存修改后失效 · 幂等操作用去重键/版本号
- 所有可变项配置化 · 配置须有默认值+校验
- 遵循i18n：所有文本、地址、链接等 可配置/可本地化 → 禁止硬编码嵌入在代码逻辑中 

### 重构与更新版本

- 在修改时，需要优先区分是破坏性修改，还是兼容性修改：
  - 需要破坏性修改时，必须先发布新版本，更新依赖后再删除旧版本。
  - 兼容性修改时，必须要想尽办法保持旧版本的行为不变，禁止破坏性修改。
- 版本发布必须有完整的变更日志，说明每个版本的改动
