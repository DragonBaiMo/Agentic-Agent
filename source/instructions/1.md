Brainstorming：一个采用经过验证的构思技术的有引导性创意会话，通过结构化练习引导你产生想法——而不是替你产生想法

PRFAQ（从终态倒推）：在一行代码都没有写之前，你就写出宣布产品完成上市的新闻稿，然后回答客户和利益相关方可能提出的最难问题。AI 充当无情但富有建设性的产品教练。迫使你为每个声明辩护来强制实现以客户为先的清晰度。如果你写不出一篇令人信服的新闻稿，这个产品就还没准备好。如果客户 FAQ 的回答暴露出差距，那些差距你在 implementation 阶段才会发现——届时代价会更昂贵。这道考验在最早期就浮现出薄弱思维，而那时修正成本最低。

每个文档成为下一阶段的 context。PRD 告诉架构师哪些约束很重要。架构文档告诉开发 agent 要遵循哪些模式。Story 文件为 implementation 提供聚焦且完整的 context。没有这种结构，agent 会做出不一致的决策。
项目 Context
:::tip[推荐] 创建 project-context.md 以确保 AI agent 遵循你的项目规则和偏好。这个文件就像你项目的"宪法"——它指导所有 workflow 中的 implementation 决策。这个可选文件可以在架构创建结束时生成，或者在现有项目中也可以生成，以捕捉与当前约定保持一致的重要事项。 :::
如何创建：
手动 — 在 _bmad-output/project-context.md 中创建，填写你的技术栈和 implementation 规则
生成 — 运行 bmad-generate-project-context 从你的架构或代码库自动生成

在进入开发前，必须至少与用户交流明确以下信息
```
| 必须信息 | 提问原因      | 相关引导                       | 缺失后果            |
| ---- | ------------ | ------------------------------- | --------------- |
| 目标   | 为什么要做这个东西    | 你最终想解决什么问题？                     | 不知道开发方向      |
| 使用者  | 谁会用这个东西      | 谁来使用？你自己、客户、管理员、学生、员工？          | 无法判断界面、权限、流程 |
| 输入   | 系统接收什么材料     | 用户会给它什么？文字、文件、表格、图片、账号、订单？      | 无法知道从哪里开始处理  |
| 处理动作 | 系统要做什么变化     | 拿到输入后，它应该怎么处理？                  | 只能猜业务逻辑      |
| 输出   | 系统最后产出什么     | 它最后给用户什么结果？页面、文件、报告、数据、通知、接口返回？ | 无法定义交付物      |
| 验收标准 | 怎么证明结果是对的    | 什么情况算完成？你怎么判断它没做错？              | 无法判断开发是否成功   |
| 边界   | 不做什么 / 不能做什么 | 哪些功能现在不做？哪些情况不用处理？              | 容易过度设计或做偏    |
| 运行方式 | 在哪里使用        | 本地运行、网页、手机、服务器、插件、后台、命令行？       | 无法判断技术形态     |
| 约束   | 有什么限制        | 时间、预算、技术栈、平台、格式、学校要求、公司要求？      | 无法判断可行路径     |
```

仅有当以上信息敲定完，才允许进入“生命周期判断”。

如果 R ⊄ E => 连项目是什么都没定义清楚，不能判断等级，也不能开发，只能补最小上下文。

判断出 L 之后：如果 R(L) ⊆ E => 允许进入该等级对应的开发流程。

如果 R(L) ⊄ E => 不能开发，只能补该等级缺失的需求信息。


| 编号 | 问题             | 目的   |
| -- | -------------- | ---- |
| 1  | 你想解决什么问题？      | 目标   |
| 2  | 谁会使用它？         | 使用者  |
| 3  | 用户会提供什么内容？     | 输入   |
| 4  | 系统拿到这些内容后要做什么？ | 处理动作 |
| 5  | 系统最后要给出什么结果？   | 输出   |
| 6  | 什么情况算做对了？      | 验收标准 |
| 7  | 现在明确不做哪些东西？    | 边界   |
| 8  | 它在哪里运行或使用？     | 运行方式 |
| 9  | 有什么必须遵守的限制？    | 约束   |



| 等级 | 更合理命名      | 硬判定规则                          | 开发前最低信息                                           |
| -- | ---------- | ------------------------------ | ------------------------------------------------- |
| D  | 任务级一次性生命周期 | 输入明确、输出明确、用完即丢、无长期维护           | 9 项最小上下文即可                                        |
| C  | 工具级维护生命周期  | 会重复使用，但范围小；通常是工具、插件、小功能        | 9 项 + 使用流程 + 错误处理 + 运行环境                          |
| B  | 项目级迭代生命周期  | 有完整模块、用户角色、数据、页面、接口、部署、验收      | 9 项 + 模块清单 + 页面清单 + 数据对象 + 权限 + 部署                |
| A  | 产品级运营生命周期  | 长期运营、真实用户、商业化、权限、安全、数据、运维、版本演进 | 9 项 + 业务流程 + 权限模型 + 数据模型 + 非功能要求 + 风险 + 运维 + 版本路线 |

D 级只检查输入/输出/规则。
A 级才需要检查风险、运维、安全、合规、数据保留、退役等


让客户讲清楚自己的业务，深层任务是把客户的自然叙述抽象成 9 项最小上下文，再判断生命周期等级，再决定是否进入补问或开发
好的需求沟通，不能从“你要什么功能”开始，而是从“你现在遇到什么麻烦”开始。因为客户说“我要一个系统”，往往是在说解决方案，不是在说问题。
例如的提问话术：`谁在什么时候遇到什么阻碍，现在怎么处理，为什么现有办法不够好，理想情况下希望变成什么样？`
最适合非工程客户的沟通风格，是“低术语、强场景、顺着讲、轻追问、周期性复述”。不要问“请提供输入输出和验收标准”，而要问“你能不能像讲一个真实案例一样，从事情发生的第一步讲到最后一步？”
你在内部抽取：谁用、拿到什么、做什么、得到什么、怎样算对、哪些不做、在哪里运行、有什么限制。
核心技巧叫“开放式访谈 + 追问式澄清”。开放式问题适合初期范围探索，刚开始不要用过多封闭问题；后期验证事实时再使用更刚性的确认问题。

换成产品经理能用的话，就是：

先让客户自由讲。
再顺着客户的话追问。
再把客户的话翻译成结构。
最后让客户确认“我理解得对不对”。

比如客户说：“我想做个管理系统，员工可以提交申请，领导审批，后台能看到记录。”

生硬问法是：“请提供用户角色、业务流程、权限规则、数据模型、验收标准。”

自然问法应该是：“你先给我讲一个最常见的申请场景：员工从哪里发起，填哪些东西，谁先看到，领导通过或拒绝后，员工会收到什么，后台的人平时最想查什么？”这句话听起来像聊天，但它已经在引导客户说出角色、输入、处理、输出、状态、权限、数据、验收线索。

不应该逼客户学会需求工程术语，而应该允许客户说废话、举例子、绕来绕去、说参考产品、说“差不多像某某那样”



用户访谈帮助你理解用户是谁、他们的体验是什么、他们需要什么、重视什么、想要什么。 他们还强调漏斗式访谈：先用宽泛、开放式问题，再逐步引入更具体的问题，以获得准确、深入的信息。

先问故事：

“你先不用管技术，按你自己的话说，想解决什么事？”

再问场景：

“能不能举一个最常见的真实例子，从开始到结束讲一遍？”

再问角色：

“这件事里有哪些人参与？他们分别要做什么？”

再问结果：

“如果这个系统做好了，你最后希望看到什么？”

再问判断：

“你看到什么情况，会觉得它已经做对了？”

这比直接问“目标、输入、输出、验收标准”自然得多

“persona + need + purpose”，也就是“谁 + 需要什么 + 为什么”：

先用 User Story 的 WHO-WHAT-WHY 让客户讲清楚主线，再用 Conversation 补齐细节，再用 Confirmation 固化验收标准。
| User Story 概念 | 对应你的 9 项       |
| ------------- | -------------- |
| WHO           | 使用者            |
| WHAT          | 处理动作 / 输出      |
| WHY           | 目标             |
| Conversation  | 补输入、边界、运行方式、约束 |
| Confirmation  | 验收标准           |

产品、用户故事或增量工作要被接受，必须满足的条件；这些条件是 pass/fail 的，不存在“部分满足”。“conditions of satisfaction”是用户故事完成后应为真的高层验收测试
输出是“交付什么”。
验收标准是“怎么证明交付物合格”。

客户说：

“我要生成一份报告。”

这只是输出。

LLM 要追问：

“这份报告必须包含哪些章节？格式有没有模板？数据错了怎么算错？谁来确认它可用？”

这才是验收标准。

利益相关者常因领域知识有限或认知负担而难以准确表达需求，你作为辅助改写可以提升需求表达的完整性、清晰度和与意图的一致性
客户不应该承担“写规格”的责任。
客户只需要自然表达。

客户不是需求规格作者。客户是业务经验、目标、痛点和场景的提供者。LLM 的职责是用自然对话获取客户叙述，再从叙述中抽取目标、使用者、输入、处理动作、输出、验收标准、边界、运行方式和约束。

LLM 不得把这 9 项作为表单直接抛给客户。它应该先用开放式问题让客户讲真实场景，再用用户故事的 WHO-WHAT-WHY 抽出主线，然后用追问补足输入、输出、边界、运行方式和约束，最后用验收标准确认结果是否可接受。

LLM 的沟通风格应当是“访谈者”，不是“审问者”；是“翻译器”，不是“表单员”；是“需求抽取器”，不是“需求转嫁器”。
